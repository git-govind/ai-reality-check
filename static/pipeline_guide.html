LLM Response Generator src/llm/response_generator.py Entry point
Sends the user's prompt to the configured language model and returns the AI response that will be evaluated by all downstream steps.
📚 Origin & Provenance
Built on top of the Ollama local inference server and the OpenAI API. Ollama allows running quantised open-weight models (Llama 3, Mistral, Phi-3, Gemma) entirely on-device with no data leaving the machine. OpenAI provides GPT-4o access as an optional cloud backend. Both backends are abstracted behind a common interface so evaluation logic is backend-agnostic.
🎯 Why It's Here
The pipeline evaluates AI-generated text quality — to do that it first needs an AI response. This step decouples generation from evaluation: the same downstream checkers run identically whether using Llama, Mistral, GPT-4o, or a pre-written test fixture. A Demo mode provides a static response when no model is installed, enabling dry-runs of the full UI without any API keys or local model setup.
⚙ How It Works
  1. Looks up the selected display name in the model registry (models/llm_registry.py) to obtain the Ollama tag or OpenAI model ID.
  2. Dispatches the prompt via the appropriate backend client (streaming or batch).
  3. Returns the raw text response string — no post-processing, no truncation — to preserve the full output for downstream analysis.
  4. In Demo mode, returns a pre-written paragraph describing transformer architectures to exercise the full UI without a live model.
Output
ai_response: str model_name: str is_demo_mode: bool

Factual Checker src/evaluation/factual_checker.py DuckDB + Wikipedia
Extracts verifiable claims from the AI response and checks each against a local structured fact store (DuckDB) and the Wikipedia API, classifying every claim as supported, contradicted, or unverified.
📚 Origin & Provenance
Follows the Retrieval-Augmented Generation (RAG) pattern, popularised by Lewis et al. (2020) "Retrieval-Augmented Generation for Knowledge-Intensive NLP Tasks" (NeurIPS 2020). The offline-first design (local DB → live API fallback) mirrors production RAG architectures that prioritise latency and data privacy.
DuckDB — in-process analytical SQL engine by Raasveldt & Mühleisen (2019). Used here as a fast local knowledge store seeded with 57 verified facts. Zero-dependency, embedded, no server process needed.
🎯 Why It's Here
LLMs hallucinate — they generate plausible-sounding but factually incorrect statements with high confidence. This step provides an external ground-truth check independent of the model being evaluated. A model cannot cheat its own factual evaluation.

The two-tier lookup (local DB first, Wikipedia second) keeps latency low for common topics while still providing broad coverage through the live encyclopedia.
⚙ How It Works
  1. Claim extraction: breaks the response into candidate factual sentences using sentence splitting and heuristic filtering (length, presence of nouns).
  2. DuckDB lookup: computes word-overlap ratio between each claim and every fact in the local DB. A ratio ≥ 0.40 classifies the claim as supported; ≥ 0.20 triggers a contradiction check against negation patterns.
  3. Wikipedia fallback: for claims not resolved by DuckDB, queries the Wikipedia search API and applies the same word-overlap logic against the returned summary. Capped at 5 claims to limit API round-trips.
  4. Weighted scoring: DuckDB support = 1.0 credit, Wikipedia = 0.9, unverified = 0.5, contradicted = 0.0 − 15 pt penalty each. Final score = weighted sum ÷ claims checked × 100.
Output
score [0–100] claims_checked supported contradicted unverified details[ ]

Consistency Checker src/evaluation/consistency_checker.py Heuristic + LLM blend
Detects internal contradictions and prompt–response mismatches using a rule-based pass plus an optional LLM self-critique, then produces a blended consistency score.
📚 Origin & Provenance
The LLM-as-Judge pattern is drawn from Zheng et al. (2023) "Judging LLM-as-a-Judge with MT-Bench and Chatbot Arena" — using a capable LLM to evaluate a weaker one's output quality.

The heuristic first-pass (6 contradiction patterns: "never … always", "on the one hand … however", absolute quantifiers, etc.) provides a fast, reproducible baseline that doesn't require any model to be running.
🎯 Why It's Here
LLMs sometimes generate responses that internally contradict themselves or stray from the original prompt. A factually accurate response can still be logically inconsistent or off-topic.

The dual-pass design balances reliability (heuristic always runs, score is deterministic and reproducible) with depth (LLM critique catches nuanced contradictions that simple pattern matching misses). The blend weights are configurable: default 30% heuristic + 70% LLM.
⚙ How It Works
  1. Heuristic pass: regex scan for 6 contradiction signal patterns. Starts at 80, deducts 10 pts per match, floor at 50. Fast, deterministic, no API call needed.
  2. LLM critique (if model configured and feature flag enabled): prompts the model with a structured critique request, extracts a X/10 score and issue bullets using regex.
  3. Score blend: final = 0.30 × heuristic + 0.70 × llm_score when LLM ran; heuristic-only otherwise.
  4. Issue bullets from both passes are deduplicated and tagged [Consistency] for display in the UI.
Output
score [0–100] heuristic_score llm_score issues[ ] critique_text

Bias & Safety Checker src/evaluation/bias_safety_checker.py Rule engine + LLM
Detects biased language and unsafe content using two independent rule sets plus an optional LLM scoring pass, producing separate bias and safety scores that are averaged into a single component score.
📚 Origin & Provenance
The bias rule vocabulary draws from AI fairness literature: Blodgett et al. (2020) "Language (Technology) is Power: A Critical Survey of 'Bias' in NLP" (ACL 2020) and the broader sociolinguistics research on loaded/pejorative terminology.

Safety categories map to established content policy frameworks: CSAM, weapons, self-harm, unqualified medical advice — categories that appear across OpenAI, Anthropic, and Meta's content policies and the EU AI Act's prohibited-use provisions.
🎯 Why It's Here
Even factually accurate, logically consistent LLM outputs can be harmful if they propagate stereotypes, use loaded language, or provide dangerous guidance. This step acts as a content safety layer independent of the factual and consistency checks.

The safety score is intentionally kept rule-only (not blended with LLM) because hard safety rules should be deterministic and not subject to LLM interpretation. Bias detection uses the LLM blend for nuance.
⚙ How It Works
  1. Bias rules (6): regex patterns for loaded qualifiers, absolute generalisations, coded language, and tribal framing. Starts at 100, deducts 20 pts per match.
  2. Safety rules (6): keyword patterns for weapons, CSAM, self-harm, unqualified medical claims, illegal-activity instructions. Deducts 25 pts per match. Pure rule-only — no LLM influence on safety score.
  3. LLM bias blend (optional, feature-flag guarded): prompts for a Bias Score: X/10, blended 40% rule + 60% LLM into final bias_score.
  4. Final component score = (bias_score + safety_score) / 2.
Output
score [0–100] bias_score safety_score bias_flags[ ] safety_flags[ ]

Clarity & Completeness Scorer src/evaluation/clarity_scorer.py Heuristic NLP
Measures whether the AI response is readable (clarity) and addresses the original prompt sufficiently (completeness), producing two sub-scores that are combined into a single clarity component.
📚 Origin & Provenance
The sentence-length readability penalty mirrors principles from the Flesch–Kincaid readability tests (Kincaid et al., 1975) — originally developed for U.S. Navy training manuals, now the most widely used automated readability formula.

The filler and vague phrase taxonomy ("As an AI language model", "it depends", "Great question!") is specific to LLM output quality assessment — a recognised category of unhelpful AI response patterns documented in RLHF training literature.
🎯 Why It's Here
A factually accurate, consistent, unbiased response is still low quality if it's incomprehensible, waffle-heavy, or fails to actually answer the question asked.

Completeness catches off-topic answers; clarity catches unreadable ones. Both are critical dimensions of LLM response quality that aren't captured by any of the other checkers.
⚙ How It Works
  1. Completeness: extracts content keywords (4+ chars, stopwords excluded) from the prompt, counts how many appear in the response. score = min(100, overlap_ratio × 130). Short responses (≤30 words) capped at 55; very long (≥600 words) flagged.
  2. Clarity — long sentences: if average sentence length > 35 words, deduct 10 pts.
  3. Clarity — vague phrases: 8 patterns ("it depends", "many factors", "in summary"). Deduct 4 pts each after the first 2, capped at −20.
  4. Clarity — filler phrases: 7 LLM-specific fillers ("As an AI", "Great question"). Deduct 5 pts each, capped at −15.
  5. Structure bonus: +5 pts if markdown headers or bullet lists detected in the response.
  6. Final blend: score = 0.60 × completeness + 0.40 × clarity.
Output
score [0–100] completeness_score clarity_score avg_sentence_len has_structure issues[ ]

Text Scoring Engine src/scoring/scoring_engine.py Weighted MCDA aggregation
Combines all five component scores into a single weighted confidence score and assigns a letter grade (A–F) with a human-readable explanation for the final verdict.
📚 Origin & Provenance
Implements Multi-Criteria Decision Analysis (MCDA) — specifically a simple weighted linear combination, the most interpretable aggregation method in decision science (Hwang & Yoon, 1981, Multiple Attribute Decision Making).

All weights and grade thresholds are externalised to config/weights.yaml and config/thresholds.yaml — no code change is needed to retune the evaluation for different use-cases (e.g. raising the accuracy weight for factual news evaluation vs. creative writing assessment).
🎯 Why It's Here
Five separate component scores are useful for diagnosis but impractical for a summary verdict. This step reduces dimensionality to a single interpretable number while preserving all component detail for drill-down.

The weighted design encodes domain knowledge: accuracy (30%) matters most for factual AI responses; safety (20%) and consistency (20%) are next; bias, clarity, and completeness each contribute equally (10%). These reflect the relative seriousness of different failure modes.
⚙ How It Works
# Weighted linear aggregation (weights loaded from config/weights.yaml) confidence_score = accuracy_score × 0.30 # factual correctness + consistency_score × 0.20 # internal logic + safety_score × 0.20 # harmful content absence + bias_score × 0.10 # language fairness + clarity_score × 0.10 # readability + completeness_score × 0.10 # prompt coverage
Accuracy30%
Consistency20%
Safety20%
Bias10%
Clarity10%
Completeness10%
Grade thresholds from config/thresholds.yaml: A ≥ 90 B ≥ 78 C ≥ 65 D ≥ 50 F < 50
Output

Metadata Checker image_evaluator/metadata_checker.py EXIF / XMP / ICC
Analyses the file's embedded metadata — EXIF tags, XMP descriptors, ICC colour profiles — to distinguish camera-captured images from AI-generated or heavily post-processed ones based on the presence, absence, and consistency of hardware-specific metadata fields.
📚 Origin & Provenance
EXIF (Exchangeable Image File Format) was standardised by JEIDA in 1995 and is now maintained by CIPA (Camera & Imaging Products Association). It defines the binary structure of metadata embedded by camera firmware into every JPEG, TIFF, and WEBP file at capture.

MakerNote (EXIF tag 37500) is a camera-manufacturer private blob written by firmware to store proprietary settings — AF data, lens corrections, custom curves. No AI generator has ever been observed embedding a MakerNote.
Key discriminator: AI generators (Midjourney, Stable Diffusion, DALL-E, Firefly) produce images with no EXIF, or embed their own software tag. Real cameras always embed make, model, timestamp, and typically MakerNote + GPS.
🎯 Why It's Here
Metadata is the fastest and cheapest authenticity signal — it requires no model, no GPU, and no pixel decode. A missing EXIF on a JPEG from a claimed camera source, or the presence of "Midjourney" in the Software tag, is definitive evidence.

Conversely, a rich authentic metadata fingerprint (MakerNote + GPS + camera pipeline) is strong positive evidence even when pixel analysis is inconclusive.
⚙ Scoring Formula (starts at 100)
−30No EXIF (JPEG)
−40AI generator software tag
−20No camera make/model
−15Power-of-two dimensions
−10Timestamp anomaly
+20Valid camera pipeline
+15MakerNote present
+10GPS coordinates
+10EXIF thumbnail
Power-of-two dimensions — widths/heights that are exact powers of two (256, 512, 768, 1024, 1536, 2048…) are a tell-tale of diffusion model generators, which operate on fixed-size latent spaces. No camera produces these dimensions. Penalty: −15 to both metadata score and editing_penalty.
Output
score [0–1] flags[ ] editing_penalty detected_ai_generator confidence_band

Pixel Forensics image_evaluator/pixel_forensics.py 5 classical techniques
Applies five complementary classical digital-forensics analyses to the raw pixel data — ELA, JPEG Ghost, noise residual, FFT spectral scan, and quantisation table inspection — each targeting a different type of authenticity artifact.
📐 Combined Pixel Score Formula
pixel_score = 100 − (ela_bad × 0.30 + ghost_bad × 0.20 + noise_bad × 0.25 # noise_bad = 60%×patch + 40%×16-block + fft_bad × 0.25)
Note: FFT peaks are included in pixel_score for general editing/artefact detection. However, they are excluded from ai_likelihood computation because JPEG 8×8 block-DCT quantisation routinely creates identical spectral peaks in authentic camera photos — making FFT unreliable as an AI-vs-real discriminator at that level.
A
Error Level Analysis (ELA) Krawetz 2007
Re-compression residual maps edited regions.
📚 Origin
Introduced by Neal Krawetz in 2007 at the Black Hat Security Conference. One of the oldest and most widely used image-forgery detection techniques. Exploits the lossy nature of JPEG compression — a region that has been saved more times has lower residual error than a freshly spliced region.
⚙ How It Works
  1. Re-compress the original image as JPEG at quality = 75.
  2. Compute per-pixel absolute difference between original and re-compressed.
  3. Measure the 95th percentile (p95) of the difference map.
  4. ela_bad = min(100, p95 / 30 × 100)
Signals ela_p95 (badness 0–100)
B
JPEG Ghost Analysis Farid 2009
Multi-quality residual inconsistency reveals splicing history.
📚 Origin
Published by Hany Farid (Dartmouth / UC Berkeley, 2009) in "Exposing Digital Forgeries in Complex Lighting Environments". Farid's group is the leading academic lab in digital-image forensics. The technique exploits the fact that different image regions have different JPEG compression quality histories — a dead giveaway for compositing.
⚙ How It Works
  1. Re-compress at 5 quality levels: [50, 60, 70, 80, 90].
  2. For each 32×32 patch, find the quality level producing the minimum residual — the patch's native quality.
  3. Build a histogram of native-quality indices across all patches.
  4. Compute normalised Shannon entropy: 0 = all patches agree (pristine); 100 = maximally spread (heavily spliced).
Signals ghost_score (entropy 0–100)
C
Noise Residual + 16-block CV PRNU – Lukas 2006
Camera sensor noise is spatially stationary; AI synthesis is not.
📚 Origin
Derived from Photo Response Non-Uniformity (PRNU) fingerprinting by Lukas, Fridrich & Goljan (2006), "Digital Camera Identification from Sensor Pattern Noise" (IEEE TIFS). PRNU is the dominant technique for camera-source attribution in digital forensics.

The 16-block macro consistency check is a custom extension: diffusion models synthesise smooth faces, textured hair, and complex backgrounds as separate region-generation passes, producing systematic spatial variation in residual noise that real PRNU never exhibits.
⚙ How It Works
  1. Subtract Gaussian-blurred image to isolate noise residual.
  2. Patch CV: compute per-patch noise std across 32×32 patches; flag if mean std < 2.0 (too smooth) or CV > 0.60 (inconsistent).
  3. 16-block CV: divide residual into 4×4 grid; compute CV of 16 block noise stds. Real cameras: low CV. AI generators: high CV.
  4. noise_bad = 60%×patch_bad + 40%×block_cv_bad
Signals noise_block_consistency (CV)
D
FFT Spectral Analysis Classical signal processing
Periodic artifacts from GAN upsampling create spectral spikes.
📚 Origin
The 2D Discrete Fourier Transform for image forensics has been used since the 1990s. Its application to GAN detection was highlighted by Durall et al. (2020) "Watch Your Up-Convolution: CNN Based Generative Deep Neural Networks Are Failing to Reproduce Spectral Distributions" (CVPR 2020).
Photo rule: FFT feeds into pixel_score (editing artefacts) but is excluded from the ai_likelihood pixel boost for photos — JPEG 8×8 block-DCT creates identical spectral peaks in authentic camera photos, making FFT an unreliable AI discriminator at that level.

Illustration exception: FFT is included in the ai_likelihood boost for illustrations (typically PNG/lossless). Without JPEG block-DCT compression, elevated FFT peaks genuinely indicate synthetic periodicity from diffusion or GAN generation.
⚙ How It Works
  1. Compute 2D power spectrum; mask the DC component (centre).
  2. Measure peak-to-mean ratio in log scale: norm_ratio = log1p(peak/mean) / log1p(1e6).
  3. fft_bad = min(100, norm_ratio / 0.25 × 100)
Signals fft_peak_ratio [0–1]
E
JPEG Quantisation Table Check JPEG standard ISO 10918
Unusual DQT table values indicate AI or non-camera origin.
📚 Origin
JPEG DQT (Define Quantisation Table) markers are part of the JPEG standard (ISO/IEC 10918-1, 1994). Every JPEG encoder embeds these tables and they encode the quality-factor profile of the compression. Camera manufacturers use well-known standardised tables. Some AI export pipelines use all-uniform tables or near-lossless tables that no camera produces.
⚙ How It Works
  1. Parse raw JFIF byte stream to locate DQT marker segments (0xFFDB).
  2. Extract all 64-coefficient quantisation tables.
  3. Flag if all 64 coefficients are identical (uniform table), or if max coefficient < 4 (near-lossless — atypical for cameras).
  4. Applies an informational −5 to pixel_score outside the main formula if suspicious.
Signals quant_suspicious: bool
Step output
score [0–1] artifacts[ ] ela_max_diff fft_peak_ratio ghost_score noise_block_consistency confidence_band

AI Artifact Classifier image_evaluator/ai_artifact_classifier.py ML + FreqNet + 6 heuristics
Estimates the probability that an image was AI-generated using three complementary methods: a HuggingFace ViT classifier (primary), a FreqNet-inspired spectral analysis, and six pixel heuristics — then applies a physics-based ML-gated noise-block boost to the final ai_likelihood.
📚 HuggingFace Model Origin
umm-maybe/AI-image-detector — a Vision Transformer (ViT) binary classifier fine-tuned on a large dataset of real photographs vs. images generated by DALL-E, Midjourney, Stable Diffusion, and other diffusion models. Published on Hugging Face Hub.

ViTs were introduced by Dosovitskiy et al. (2020) "An Image is Worth 16×16 Words: Transformers for Image Recognition at Scale" (ICLR 2021). Fine-tuning for AI-generated image detection was pioneered by several groups in 2022–2023 as diffusion models became widespread.
FreqNet: inspired by Wang et al. (2020) "CNN-Generated Images Are Surprisingly Easy to Spot…For Now" (CVPR 2020). Upsampling in generative networks introduces detectable high-frequency artifacts in the frequency domain.
🎯 Why It's Here
Classical forensics (ELA, FFT) detects editing artifacts but were designed before diffusion models existed. An ML classifier trained specifically on real-vs-AI-generated images captures the subtle, global statistical patterns that diffusion models imprint — patterns that are invisible to classical techniques.

The three-tier ensemble (ML + frequency + heuristics) provides redundancy: if the ML model is uncertain or OOD, the FreqNet and heuristics provide independent supporting signals.
⚙ Ensemble Combination + Type-Aware ML-Gated Pixel Boost
# Combine ML + heuristics into raw ai_prob if hf_model_available: ai_prob = 0.80 × hf_model_prob + 0.20 × heuristic_blend else: ai_prob = 0.70 × heuristic_mean + 0.30 × freqnet_prob # Type-aware gate selection (image_scoring.py) if image_type == "illustration": gate_low, gate_high, cap = 0.00, 0.15, 0.60 fft_sig = clip((fft_ratio − 0.30) / 0.25, 0, 1) # FFT included (PNG/lossless) pixel_sig = (fft_sig + consist_sig) / 2 elif image_type == "photo" and no_exif and (block_cv ≥ 0.45 or ela_bad ≥ 20): gate_low, gate_high, cap = 0.00, 0.15, 0.60 # conditional: no_exif + pixel corroboration pixel_sig = clip((block_cv − 0.15) / 0.40, 0, 1) # FFT excluded for photos else: gate_low, gate_high, cap = 0.10, 0.30, 0.50 # standard photo gate pixel_sig = clip((block_cv − 0.15) / 0.40, 0, 1) gate = clip((ai_prob − gate_low) / (gate_high − gate_low), 0, 1) ai_prob* = ai_prob + gate × pixel_sig × cap × (1 − ai_prob)
Gate rationale: The standard photo gate (gate_low=0.10) prevents JPEG block-DCT artefacts from boosting ai_likelihood when the ML model is confident the image is real. Two additional gate paths handle cases where the ML classifier under-detects:

• Illustration gate (gate_low=0): ML models trained on photorealistic deepfakes fail on cartoon/rendered AI art. FFT is included because illustrations are PNG/lossless.
• Photo conditional gate (gate_low=0, triggers on no_exif + high block_cv or ELA): handles photorealistic AI images where no_exif and spatially inconsistent noise or synthetic-pixel JPEG compression patterns corroborate the AI hypothesis.
6 pixel heuristics: texture smoothness · colour entropy · edge sharpness · high/low frequency ratio · saturation distribution skew · bilateral symmetry score
Output

Image-Text Consistency (CLIP) image_evaluator/image_text_consistency.py OpenAI CLIP 2021
Measures semantic alignment between the image and its accompanying caption using CLIP cosine similarity — detecting mismatched or misleading captions added to authentic or AI-generated images.
📚 Origin & Provenance
CLIP (Contrastive Language–Image Pre-Training) was published by Radford et al. (OpenAI, 2021) in "Learning Transferable Visual Models from Natural Language Supervision" (ICML 2021). Trained on 400 million image–text pairs scraped from the internet, CLIP maps images and text into a shared semantic embedding space — enabling zero-shot similarity measurement without any task-specific fine-tuning.
Model used: clip-ViT-B/32 via the sentence-transformers library (Reimers & Gurevych, 2019). Cosine similarity is mapped from the empirical [0.15–0.35] range for photo–caption pairs to a [0, 1] consistency score.
🎯 Why It's Here
A common manipulation tactic is context injection: adding a misleading caption to an authentic photograph to change its perceived meaning (e.g., labelling a protest photo from one country as being from another).

CLIP detects this: if the image and caption describe semantically different content, the cosine similarity is low. This step runs only when a caption is provided; without a caption its weight is redistributed to other components.
⚙ How It Works
  1. CLIP path (preferred): encode image and caption independently using clip-ViT-B/32; compute cosine similarity. Map from [0.15, 0.35] → [0, 1].
  2. Sub-phrase re-ranking: if full-caption similarity is low, re-test individual noun phrases from the caption to identify which specific concepts are inconsistent.
  3. Keyword fallback (no CLIP installed): extract dominant colours and nouns from the image description; check keyword overlap with the caption. Less precise but dependency-free.
  4. Step is skipped entirely (weight redistributed) when no caption is supplied.
Output
score [0–1] method (clip/keyword) consistency_issues[ ] ran: bool

Reverse Image Search image_evaluator/reverse_image_search.py pHash + 3 APIs
Queries web image indexes to check whether the image (or a near-identical copy) has appeared online before, verifying provenance and detecting recycled or recontextualised images.
📚 Origin & Provenance
Reverse image search was pioneered by Google Images (2011) and is now offered by Google Cloud Vision API, Microsoft Bing Visual Search, and third-party aggregators like SerpApi (Google Lens backend).

Perceptual hashing (pHash) — used for local refinement — was introduced by Zauner (2010) in "Implementation and Benchmarking of Perceptual Image Hash Functions". pHash captures structural similarity that survives JPEG re-compression, scaling, and minor colour adjustments.
🎯 Why It's Here
A reverse image search hit establishes prior publication — evidence the image existed before the claimed context. High similarity matches indicate the image is a copy or derivative, not an original.

This step runs only when an API key is configured; without one, its 10% weight is redistributed to the remaining active components. The three-backend priority order (Google → Bing → SerpApi) maximises hit rate while remaining cost-efficient.
⚙ How It Works
  1. Try backends in priority order: Google Cloud Vision API → Bing Visual Search API → SerpApi (Google Lens). First successful response wins.
  2. For each returned source URL, attempt to download the image and compute pHash distance for refined similarity.
  3. Map best similarity score to one of four tiers: ≥0.90 → 1.00 (exact match), ≥0.70 → 0.80, ≥0.30 → 0.50, otherwise 0.00.
  4. Step skipped if no API key is configured — weight redistributed proportionally among active steps.
Output
score [0–1] found: bool similarity source_urls[ ] ran: bool

Image Scoring Engine image_evaluator/image_scoring.py Adaptive MCDA + confidence bands
Aggregates all five pipeline results into a single weighted authenticity score using image-type-adaptive weights, dynamic redistribution for skipped steps, and per-component confidence bands — then produces the final grade, top signals, and human-readable explanation.
📚 Origin & Provenance
Again uses MCDA weighted linear combination. The innovation here over the text pipeline is adaptive weights by image type: a photo, an illustration, and a screenshot each have different reliability profiles for each detector (e.g., the AI classifier's weights are lower for illustrations because the training data is primarily photographs).

Confidence band propagation follows the uncertainty quantification principle: each component reports a ±band reflecting how reliable its score is, and these are propagated to the final authenticity score as Σ(w_i × band_i).
🎯 Why It's Here
A single composite score is needed for the grade and summary. The adaptive weights make the system honest: it doesn't apply photo-centric heuristics with full weight to an artwork or a screenshot where those heuristics have lower discriminative power.

The missing-step redistribution (normalising weights when consistency or reverse search is skipped) ensures the score is always calibrated to the same 0–100 scale regardless of which optional steps ran.
⚙ How It Works
# 1. Detect image type from pixel signals # flat_ratio>0.30 or unique_ratio<0.10 → screenshot # noise_std>2.5 and flat_ratio<0.25 → photo (cap=0.25: snow/sky/fog tolerance) # else → illustration image_type = detect_type(flat_ratio, unique_colour_ratio, noise_std) # 2. Select weight preset (from config/weights.yaml) base_w = WEIGHTS_BY_TYPE[image_type] # 3. Compute type-aware ML-gated effective ai_prob (see AI Classifier card) ai_prob* = ai_prob + gate × pixel_sig × cap × (1 − ai_prob) # 3b. Watermark floor — confirmed AI watermark overrides pixel/ML verdict if watermark and watermark.has_watermark: ai_prob* = max(ai_prob*, 0.85) # 4. Build scores dict, redistribute weights for skipped steps scores = { metadata, pixel, ai: 1−ai_prob*, [consistency], [reverse] } norm_w = normalize_weights(active_weights) # 5. Weighted sum → authenticity_score auth = sum(scores[k] × norm_w[k] for k in norm_w) × 100 # 6. Editing likelihood (formula-based) editing = 0.60 × ela_badness + 0.40 × metadata_edit_penalty
